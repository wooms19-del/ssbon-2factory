# 웹 → 품목 마스터 기반 전환 계획

목표: 지금 흩어진 설정 데이터(L.products / L.recipes / L.sauces / L.submats)를 걷어내고,
**품목 마스터(item_master / item_recipe / external_key_map)를 유일한 원본**으로 삼는다.
마스터가 제품·레시피·소스·부재료를 뿌려주고, 화면들은 받아서 쓰기만 한다.

핵심 전략: **dataLayer.js(DL)를 허브로.**
이미 있는 DL을 완성해서, 마스터를 읽어 기존 형식(L.products/L.recipes)으로 자동 생성.
→ 화면 14개는 코드를 안 바꿔도 됨. 형식은 그대로, 내용만 마스터에서 나옴.

## 1. 지금 뭐가 뭘 참고하나 (전수조사 결과)

### L.products (제품 목록) — 14개 파일이 사용
| 참고 필드 | 용도 | 쓰는 파일 |
|---|---|---|
| kgea (개당 원육 kg) | 원육 투입량·수율 계산 | analysis, daily_report, dashboard, monthly_production, packing, performance |
| name (제품명) | 제품 선택 드롭다운 | packing, dashboard, settings, shipment, timetable |
| noMeat (고기 없음 플래그) | 메추리알 등 예외 처리 | inedible_prod, analysis, timetable, monthly_production |
| capa (생산능력) | 생산성 분석·대시보드 | analysis, dashboard |
| sauce (소스 종류) | 대시보드 표시 | dashboard |

### L.recipes (레시피) — packing / outerpacking 사용
| 참고 필드 | 용도 | 쓰는 파일 |
|---|---|---|
| outer (포장재 목록·수량) | **외포장 박스·트레이 수량 계산, perBox(박스당 입수)** | outerpacking (4곳) |
| inner (내포장 재료) | 부재료(버터·메추리알) 필요 판정 | packing |

### L.sauces / L.submats — dashboard / packing / trace 사용
소스 2종, 부재료 2종. 표시·선택용.

## 2. 마스터로 어디서 오나 (전환 매핑)

| 기존 데이터 | 마스터 출처 | 전환 방법 | 상태 |
|---|---|---|---|
| 제품명 (name) | external_key_map.webName | 매핑 11종 그대로 | 마스터에서 |
| 개당 원육 (kgea) | item_recipe 반제품 inner의 파쇄육 kg | 레시피에서 유도 (기존값과 일치 확인됨) | 마스터에서 |
| 소스 (sauce) | item_recipe inner의 소스 코드 | 레시피에서 유도 | 마스터에서 |
| 고기없음 (noMeat) | item_recipe inner에 파쇄육 없으면 true | 레시피에서 유도 (메추리알 180g만 해당) | 마스터에서 |
| 포장재 (outer) | item_recipe 완제품 outer | 마스터에서 (⚠ 단위 주의) | 마스터에서 |
| 부재료 (inner) | item_recipe 반제품 inner | 마스터에서 | 마스터에서 |
| **생산능력 (capa)** | **마스터에 없음** | **현장값으로 별도 보관** | 유지 |

### ⚠ 단위 주의 (outer / perBox)
- 기존 L.recipes는 **낱개(1개) 기준**: 코스트코 인박스 0.25 → perBox 4
- 마스터 완제품은 **세트(4입·5입) 기준**: 인박스 1
- 외포장 계산(perBox)이 여기에 걸림 → 변환 시 세트→낱개 환산 필요
- 이 부분만 검증을 거쳐 전환. 나머지(kgea/부재료)는 안전.

## 3. 전환 단계 (안전 순서)

1. **[완료] 마스터 데이터 구축** — item_master 107 / item_recipe 51 / external_key_map 11 (Firestore)
2. **[완료] 마스터 조회 화면** — 레시피 탭에서 조회
3. **DL이 마스터 로드** — 앱 시작 시 마스터 3종 로드
4. **DL이 마스터 → L.products 생성** — name/kgea/sauce/noMeat (capa는 별도 맵 유지)
5. **DL이 마스터 → L.recipes 생성** — inner/outer (단위 환산 검증)
6. **화면별 대조 검증** — 14개 화면이 기존과 같은 값 나오는지 확인
7. **설정 손등록 UI 제거** — 제품/소스/부재료/레시피 관리 걷어냄
8. **capa만 남긴 최소 설정** — 생산능력만 제품별로 관리

핵심: 3~5는 기존 형식(L.products/L.recipes)을 마스터에서 채우는 어댑터.
→ 화면 코드는 안 바뀜. 깨질 위험 최소. 검증만 철저히.

---

## 4. 검증 기준 (각 단계)

- kgea: 마스터 유도값 == 기존 L.products.kgea (제품 8종 전부 일치해야)
- 수율: 전환 전후 일별실적·월단위생산량 수율이 동일해야
- 외포장: 박스·트레이·perBox가 전환 전후 동일해야 (단위 환산 정확)
- 다중 디바이스: 모든 기기에서 같은 값
